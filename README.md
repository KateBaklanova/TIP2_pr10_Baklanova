## Теория

Горизонтальное масштабирование означает увеличение мощности системы не за счёт «усиления» одного сервера, а за счёт добавления нескольких экземпляров одного и того же приложения. Именно эта логика заложена и в вашем черновике: вместо одного экземпляра tasks запускаются tasks_1, tasks_2, tasks_3, а балансировщик распределяет между ними запросы.

Если один экземпляр уже не справляется с нагрузкой, есть два пути:

- вертикально увеличить ресурсы одной машины;
- горизонтально поднять несколько экземпляров сервиса.

Для современных backend-систем второй путь часто оказывается более гибким, особенно если приложение уже упаковано в контейнер и может быстро запускаться в нескольких копиях.

Если поднять несколько реплик сервиса, этого ещё недостаточно. Нужен компонент, который будет принимать клиентские запросы и направлять их на один из доступных экземпляров приложения.
Именно эту роль выполняет load balancer.

NGINX — это один из самых распространённых reverse proxy и балансировщиков нагрузки. Он удобен для учебной практики, потому что:

- легко запускается в контейнере;
- конфигурация достаточно понятна;
- можно быстро описать upstream;
- можно прокидывать заголовки;
- можно наглядно увидеть распределение трафика.

В рамках этой практической работы NGINX используется как учебный слой между клиентом и репликами сервиса.

### Контрольные вопросы

1. Что такое горизонтальное масштабирование?

Увеличение количества экземпляров сервиса 

2. Чем оно отличается от вертикального масштабирования?
Вертикальное - увеличение мощности одного сервера (CPU/RAM), горизонтальное - добавление новых серверов/контейнеров.

3. Зачем нужен load balancer?

Чтобы распределять входящие запросы между несколькими репликами сервиса

4. Какую роль в этой работе выполняет NGINX?

Роль reverse proxy и балансировщика нагрузки между репликами tasks

5. Что такое upstream в NGINX?

Группа backend-серверов (реплик), между которыми NGINX распределяет трафик.

6. Почему для горизонтального масштабирования желательно делать сервис stateless?

Чтобы любая реплика могла обработать любой запрос без потери данных о состоянии пользователя.

7. Зачем нужен health endpoint?

Для проверки доступности и работоспособности каждого экземпляра сервиса

8. Почему полезно добавлять X-Instance-ID?

Чтобы наглядно видеть, какая конкретная реплика обработала запрос.

9. Что происходит при остановке одной из реплик?

Балансировщик перестаёт направлять запросы на неё, система продолжает работать на оставшихся репликах

10. Почему клиенту удобнее работать с одной точкой входа, а не с несколькими адресами реплик?

Клиенту не нужно знать внутреннюю структуру и адреса всех реплик, достаточно одного адреса балансировщика
